## Visão geral

O firewall principal da **Tekz Tecnologias** é um **OPNsense** virtualizado em um appliance VMware.

Ele é o ponto central da rede interna e é responsável por:

- gateway da LAN principal;
- DHCP da LAN e VLANs;
- roteamento interno;
- isolamento entre redes;
- regras de firewall;
- redirecionamentos de porta;
- publicação de serviços internos;
- servidor OpenVPN para acesso externo seguro.

## Isolamento entre VLANs

Todas as VLANs possuem regras no firewall bloqueando a comunicação direta entre uma rede e outra.

Essa política reduz o risco de:

- visitantes acessarem a rede interna;
- equipamentos de bancada alcançarem servidores sem necessidade;
- redes específicas acessarem serviços administrativos;
- movimentação lateral entre redes em caso de incidente.

## Política esperada entre redes

| Origem | Destino | Status esperado |
| --- | --- | --- |
| LAN principal | Internet | Permitido |
| VLAN 100 / Bancada | Internet | Permitido conforme regra |
| VLAN 11 / Rider | Internet | Permitido conforme regra |
| VLAN 10 / Visitantes | Internet | Permitido conforme regra |
| VLAN 10 / Visitantes | LAN principal | Bloqueado |
| VLAN 100 / Bancada | LAN principal | Bloqueado ou restrito |
| VLAN 11 / Rider | LAN principal | Bloqueado ou restrito |
| VLANs entre si | Outras VLANs | Bloqueado |

<Warning>
  Qualquer liberação entre VLANs deve ser feita com regra específica, informando origem, destino, porta, protocolo e justificativa.
</Warning>

| Item | Informação |
| --- | --- |
| Serviço | OpenVPN |
| Protocolo | UDP |
| Porta | `1194` |
| Rede VPN | `10.20.30.0/24` |
| Finalidade | Acesso externo seguro à rede interna da Tekz |

## Uso da VPN

A VPN deve ser utilizada para acessar serviços privados da infraestrutura interna, como:

- OPNsense;
- Proxmox;
- Passbolt;
- Services;
- Univention Server;
- Nextcloud interno;
- VM Automation;
- painéis e serviços não publicados publicamente.

## Publicação de serviços

A publicação principal dos serviços internos ocorre através de NAT no OPNsense, encaminhando as portas `80` e `443` para a VM `services`, onde o Traefik faz o roteamento por domínio.

Fluxo padrão:

```text
Internet
    ↓
Cloudflare
    ↓
IP público local da Tekz
    ↓
OPNsense
    ↓
NAT 80/443
    ↓
VM services - 172.16.0.253
    ↓
Traefik
    ↓
Container / serviço correspondente
```

## Regras administrativas locais

As regras abaixo aparecem na LAN para acesso administrativo ao próprio firewall.

| Interface | Protocolo | Destino | Porta | Descrição |
| --- | --- | --- | --- | --- |
| LAN | TCP | LAN address | `443` | Anti-Lockout Rule |
| LAN | TCP | LAN address | `80` | Anti-Lockout Rule |
| LAN | TCP | LAN address | `22` | Anti-Lockout Rule |

<Note>
  As regras Anti-Lockout evitam perda de acesso administrativo ao firewall pela LAN.
</Note>

## Recomendações de segurança

- Priorizar publicação de serviços via Traefik.
- Evitar exposição direta de painéis administrativos na WAN.
- Revisar NATs antigos da VM `automation`.
- Validar se Elastic precisa continuar exposto.
- Validar se n8n legado ainda é necessário.
- Validar se Evolution API antigo ainda é necessário.
- Restringir serviços administrativos por VPN sempre que possível.
- Manter regras com descrição clara.
- Registrar alterações relevantes em `infra-tekz/incidentes`.
- Fazer backup da configuração do OPNsense antes de alterações importantes.

## Checklist antes de alterar regras

Antes de alterar NAT, firewall, VLAN ou VPN:

- confirmar serviço afetado;
- confirmar IP interno correto;
- confirmar porta interna e externa;
- verificar se existe dependência em Cloudflare ou Nginx Proxy Manager;
- validar se o serviço está em produção ou legado;
- registrar configuração atual;
- planejar rollback;
- testar após alteração.

## Checklist de troubleshooting

### Serviço público fora do ar

1. Verificar se o domínio resolve corretamente no Cloudflare.
2. Confirmar se o registro aponta para `managerncst.tekz.com.br` ou IP correto.
3. Validar se o IP público local está acessível.
4. Verificar NAT no OPNsense.
5. Validar se a VM de destino está online.
6. Validar se o Traefik está rodando, quando aplicável.
7. Validar se o container/serviço está ativo.
8. Conferir logs do Traefik, Portainer ou aplicação.

### VPN sem conectar

1. Confirmar se o link local da Tekz está ativo.
2. Verificar se a porta UDP `1194` está acessível.
3. Validar status do servidor OpenVPN no OPNsense.
4. Verificar certificado/configuração do usuário.
5. Conferir logs do OpenVPN.
6. Testar acesso à rede `10.20.30.0/24`.
7. Testar ping para serviços internos permitidos.

### VLAN sem internet

1. Confirmar DHCP da VLAN.
2. Confirmar gateway da VLAN.
3. Verificar regra de firewall permitindo saída para internet.
4. Validar NAT de saída, se aplicável.
5. Testar DNS.
6. Testar rota para internet.
7. Conferir se a VLAN está chegando corretamente no switch/AP.

## Pontos a confirmar

- Interfaces físicas/virtuais exatas associadas à LAN e VLANs.
- Gateway exato de cada VLAN.
- Status atual da regra DNS para `172.16.0.153`.
- Se todos os NATs legados ainda estão em uso.
- Se o Proxmox deve continuar exposto pela WAN.
- Se o Elastic deve continuar exposto pela WAN.
- Se n8n, Evolution e Chatwoot legados devem ser removidos ou migrados.
- Se o acesso administrativo deve ser limitado exclusivamente por VPN.
- Se há backup atualizado da configuração do OPNsense.

## Observações

<Note>
  O firewall OPNsense é um dos componentes mais críticos da infraestrutura interna da Tekz. Alterações incorretas podem impactar internet, VPN, publicação de serviços, VLANs, acesso remoto e operação dos sistemas internos.
</Note>
